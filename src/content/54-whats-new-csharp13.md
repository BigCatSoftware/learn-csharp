# What's New in C# 13

C# 13, released with .NET 9 in November 2024, refined several existing features and introduced new capabilities for performance, threading, and partial type members. This release continued the trend of making C# more expressive while improving low-level control.

---

## Params Collections

The `params` keyword is no longer limited to arrays. It now works with `ReadOnlySpan<T>`, `IEnumerable<T>`, and other collection types, enabling allocation-free variadic methods.

```csharp
// C# 12 — params only works with arrays (always allocates)
void LogOld(params string[] messages)
{
    foreach (var msg in messages)
        Console.WriteLine(msg);
}

// C# 13 — params with ReadOnlySpan<T> (stack-allocated, zero heap allocation)
void Log(params ReadOnlySpan<string> messages)
{
    foreach (var msg in messages)
        Console.WriteLine(msg);
}

// C# 13 — params with IEnumerable<T>
void Process(params IEnumerable<int> values)
{
    foreach (var v in values)
        Console.Write($"{v} ");
}

// All calling syntaxes still work
Log("Starting", "Processing", "Done");
Process(1, 2, 3, 4, 5);
```

### Performance Comparison

| `params` Type | Heap Allocation | Stack Friendly | Existing Collection Passable |
|---|---|---|---|
| `params T[]` | Yes (always) | No | Yes (if array) |
| `params ReadOnlySpan<T>` | No | Yes | Yes (via span) |
| `params IEnumerable<T>` | Depends | No | Yes (any enumerable) |
| `params List<T>` | Yes | No | Yes (if list) |

```csharp
// Real-world: high-performance string building
public static string Join(string separator, params ReadOnlySpan<string> values)
{
    if (values.Length == 0) return string.Empty;
    if (values.Length == 1) return values[0];

    var builder = new DefaultInterpolatedStringHandler();
    builder.AppendFormatted(values[0]);
    for (int i = 1; i < values.Length; i++)
    {
        builder.AppendFormatted(separator);
        builder.AppendFormatted(values[i]);
    }
    return builder.ToStringAndClear();
}
```

> **Tip:** Prefer `params ReadOnlySpan<T>` over `params T[]` in new code when performance matters. The compiler can stack-allocate the arguments, avoiding GC pressure entirely.

## New `Lock` Type (`System.Threading.Lock`)

A purpose-built lock type that replaces the pattern of locking on arbitrary `object` instances.

```csharp
// C# 12 — lock on an object (relies on Monitor under the hood)
public class CounterOld
{
    private readonly object _lock = new();
    private int _count;

    public void Increment()
    {
        lock (_lock)
        {
            _count++;
        }
    }
}

// C# 13 — dedicated Lock type
public class Counter
{
    private readonly Lock _lock = new();
    private int _count;

    public void Increment()
    {
        lock (_lock)
        {
            _count++;
        }
    }

    // Also supports scoped usage for advanced patterns
    public void IncrementWithScope()
    {
        using (_lock.EnterScope())
        {
            _count++;
        }
    }
}
```

> **Important:** The `lock` statement now recognizes `System.Threading.Lock` and uses its optimized `EnterScope()` method instead of `Monitor.Enter`. This produces more efficient code and provides better semantics.

| Aspect | `lock(object)` | `lock(Lock)` |
|---|---|---|
| Underlying mechanism | `Monitor.Enter`/`Exit` | `Lock.EnterScope()` |
| Type safety | Any object (error-prone) | Dedicated lock type |
| Can accidentally lock on `this` | Yes | No |
| Scoped API | No | Yes (`EnterScope()`) |
| Performance | Good | Better (optimized path) |

> **Warning:** Never lock on publicly accessible objects, strings, or `this`. The new `Lock` type makes this mistake structurally impossible.

## Partial Properties

Building on partial methods (which existed since C# 3), C# 13 adds partial properties. One part declares the signature, the other provides the implementation.

```csharp
// Declaring part (often generated by a source generator)
public partial class ViewModel
{
    public partial string Name { get; set; }
    public partial int Age { get; set; }
}

// Implementing part (often hand-written)
public partial class ViewModel : INotifyPropertyChanged
{
    private string _name = "";
    private int _age;

    public event PropertyChangedEventHandler? PropertyChanged;

    public partial string Name
    {
        get => _name;
        set
        {
            if (_name != value)
            {
                _name = value;
                PropertyChanged?.Invoke(this, new PropertyChangedEventArgs(nameof(Name)));
            }
        }
    }

    public partial int Age
    {
        get => _age;
        set
        {
            if (_age != value)
            {
                _age = value;
                PropertyChanged?.Invoke(this, new PropertyChangedEventArgs(nameof(Age)));
            }
        }
    }
}
```

> **Tip:** Partial properties shine with source generators like the MVVM Community Toolkit. The generator declares the partial property, and you implement it with custom notification logic — or vice versa.

## Method Group Natural Type Improvements

The compiler is now smarter about inferring the delegate type of method groups, reducing cases where you need explicit casts.

```csharp
// C# 12 — sometimes required explicit delegate type
// Func<int, string> converter = Convert;  // might fail with overloads
// string Convert(int x) => x.ToString();

// C# 13 — better overload resolution for method groups
var items = new List<int> { 1, 2, 3, 4, 5 };

// The compiler correctly resolves the method group
var strings = items.Select(int.ToString);  // was sometimes ambiguous
var parsed = inputs.Select(int.Parse);     // improved resolution
```

```csharp
// Practical: event handlers
button.Click += HandleClick;  // improved inference

void HandleClick(object? sender, EventArgs e)
{
    // handle click
}
```

> **Note:** This is primarily a compiler improvement — your existing code does not change, but previously ambiguous method group conversions now compile without explicit casts.

## Implicit Indexer Access in Object Initializers

You can now use the `^` (index from end) operator inside object initializers.

```csharp
// C# 12 — could not use ^ in initializers
var buffer = new int[5];
buffer[^1] = 99; // separate statement

// C# 13 — ^ works in initializers
var timer = new TimerSettings
{
    Intervals =
    {
        [0] = TimeSpan.FromSeconds(1),
        [1] = TimeSpan.FromSeconds(2),
        [^1] = TimeSpan.FromSeconds(30)  // last element
    }
};
```

```csharp
// Works with collection initializers too
var data = new DataBuffer
{
    [0] = 0xFF,
    [1] = 0x00,
    [^2] = 0xAB,
    [^1] = 0xCD
};
```

## Ref Struct Implementing Interfaces

Ref structs can now implement interfaces, unlocking generic programming scenarios for stack-only types.

```csharp
// C# 12 — ref structs could NOT implement interfaces
// C# 13 — now they can
public ref struct SpanWriter : IDisposable
{
    private Span<byte> _buffer;
    private int _position;

    public SpanWriter(Span<byte> buffer)
    {
        _buffer = buffer;
        _position = 0;
    }

    public void WriteByte(byte value)
    {
        _buffer[_position++] = value;
    }

    public void Dispose()
    {
        // cleanup if needed
        _position = 0;
    }
}

// Can now use 'using' pattern
Span<byte> buffer = stackalloc byte[256];
using var writer = new SpanWriter(buffer);
writer.WriteByte(0x01);
writer.WriteByte(0x02);
```

> **Important:** Even though ref structs can implement interfaces, they still cannot be boxed. You cannot cast a ref struct to its interface type — the interface constraint must be used with generics that have the `allows ref struct` anti-constraint.

## The `allows ref struct` Anti-Constraint

A new generic constraint that permits ref struct types as type arguments.

```csharp
// C# 12 — generic methods could not accept ref structs
// void Process<T>(T value) { } // cannot pass Span<int>

// C# 13 — allows ref struct opens the door
public static void Process<T>(T value) where T : allows ref struct, IDisposable
{
    using (value)
    {
        // work with value
    }
}

// Now works with both regular types and ref structs
Process(new MemoryStream());       // regular IDisposable
Process(new SpanWriter(buffer));   // ref struct IDisposable
```

```csharp
// Practical: generic enumeration over stack-only types
public static int Count<TEnumerator, T>(TEnumerator enumerator)
    where TEnumerator : allows ref struct, IEnumerator<T>
{
    int count = 0;
    while (enumerator.MoveNext())
        count++;
    return count;
}
```

> **Note:** `allows ref struct` is called an "anti-constraint" because it *relaxes* a restriction rather than adding one. By default, generic type parameters do not allow ref struct types.

## Overload Resolution Priority

The `[OverloadResolutionPriority]` attribute lets library authors guide the compiler to prefer newer overloads without breaking existing callers.

```csharp
public class Logger
{
    // Original method — kept for binary compatibility
    public void Log(params string[] messages)
    {
        foreach (var msg in messages)
            Console.WriteLine(msg);
    }

    // New preferred overload — better performance
    [OverloadResolutionPriority(1)]
    public void Log(params ReadOnlySpan<string> messages)
    {
        foreach (var msg in messages)
            Console.WriteLine(msg);
    }
}

var logger = new Logger();
logger.Log("Hello", "World"); // calls the ReadOnlySpan overload (higher priority)
```

| Priority Value | Meaning |
|---|---|
| 0 (default) | Normal resolution rules apply |
| Positive (1, 2, ...) | Preferred over lower-priority overloads |
| Negative (-1, -2, ...) | Deprioritized (useful for legacy overloads) |

> **Caution:** This attribute is designed for library authors managing API evolution. Application code rarely needs it. Overusing it can make overload resolution confusing to callers.

## Summary

| Feature | Primary Benefit |
|---|---|
| Params collections | Allocation-free variadic methods with `ReadOnlySpan<T>` |
| `System.Threading.Lock` | Type-safe, optimized locking primitive |

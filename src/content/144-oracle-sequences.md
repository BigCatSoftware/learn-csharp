# Oracle Sequences and Identity Columns

*Chapter 10.5 — Oracle SQL for Data Engineers*

---

## Overview

Primary key generation is a fundamental concern in database design, and Oracle and SQL
Server approach it very differently. Oracle has used **sequences** — standalone database
objects that generate unique numbers — since Oracle 8. SQL Server has used **IDENTITY**
columns since its earliest versions. Starting with Oracle 12c (2013), Oracle added
identity columns, and SQL Server 2012 added sequence objects, so modern versions of both
engines support both approaches.

As a Data Engineer migrating CMiC from Oracle to SQL Server, you need to understand how
Oracle sequences work, how to map them to SQL Server IDENTITY or SEQUENCE objects, and
how to handle the gaps and caching behaviors that affect data integrity and performance.

## Core Concepts

### 1. Oracle Sequences — CREATE SEQUENCE

A sequence is an independent schema object that generates a series of numbers:

```sql
CREATE SEQUENCE cmic.job_cost_seq
    START WITH   1
    INCREMENT BY 1
    MINVALUE     1
    MAXVALUE     999999999
    CACHE        20
    NOCYCLE
    NOORDER;
```

Key options:
- `START WITH` — first value returned
- `INCREMENT BY` — step size (can be negative for descending)
- `CACHE n` — pre-allocates n values in memory for performance
- `NOCACHE` — generates one at a time (slower but no gaps on restart)
- `CYCLE` / `NOCYCLE` — whether to wrap around at max
- `ORDER` / `NOORDER` — guarantees strict ordering in RAC environments

### 2. NEXTVAL and CURRVAL

```sql
-- Get the next value (increments the sequence)
SELECT cmic.job_cost_seq.NEXTVAL FROM DUAL;

-- Get the current value (does not increment; only valid after NEXTVAL in same session)
SELECT cmic.job_cost_seq.CURRVAL FROM DUAL;
```

**CURRVAL rules**:
- Only valid after you have called NEXTVAL at least once in the current session
- Returns the last value generated by NEXTVAL in *this* session
- Does NOT show the last value generated by another session

### 3. Using Sequences in INSERT Statements

```sql
-- Oracle: typical pattern
INSERT INTO cmic.job_cost_detail (entry_id, job_id, cost_code, amount)
VALUES (cmic.job_cost_seq.NEXTVAL, 'J-1001', '03-100', 5000.00);

-- Using CURRVAL for a child table in the same transaction
INSERT INTO cmic.job_cost_notes (note_id, entry_id, note_text)
VALUES (cmic.notes_seq.NEXTVAL, cmic.job_cost_seq.CURRVAL, 'Concrete pour');
```

### 4. Sequence Caching and Gaps

When `CACHE 20` is set, Oracle pre-allocates 20 values in shared memory. This means:

- **Performance**: NEXTVAL does not need a disk write for every call
- **Gaps**: If the instance crashes or is restarted, cached values are lost
- **Example**: Values 1–20 are cached. You use 1–7, then the instance restarts. Next
  NEXTVAL returns 21, not 8. Values 8–20 are "lost."

Gaps also occur from:
- Rolled-back transactions (the sequence does not roll back)
- RAC environments where each instance has its own cache range
- Application logic that calls NEXTVAL but does not INSERT

**Gaps are normal and expected.** Do not design systems that require gapless sequences.

### 5. Oracle 12c+ Identity Columns

Oracle 12c introduced identity columns, similar to SQL Server's IDENTITY:

```sql
-- GENERATED ALWAYS: Oracle always generates the value; cannot INSERT manually
CREATE TABLE cmic.daily_reports (
    report_id NUMBER GENERATED ALWAYS AS IDENTITY (
        START WITH 1
        INCREMENT BY 1
        CACHE 20
    ),
    job_id    VARCHAR2(20),
    report_date DATE
);

-- GENERATED BY DEFAULT: Oracle generates if you don't supply a value
CREATE TABLE cmic.equipment_log (
    log_id NUMBER GENERATED BY DEFAULT AS IDENTITY (
        START WITH 1000
        INCREMENT BY 1
        CACHE 20
    ),
    equip_id VARCHAR2(20),
    action   VARCHAR2(10)
);

-- GENERATED BY DEFAULT ON NULL: generates if NULL is supplied
CREATE TABLE cmic.timecard_entries (
    entry_id NUMBER GENERATED BY DEFAULT ON NULL AS IDENTITY,
    employee_id NUMBER,
    hours_worked NUMBER(4,2)
);
```

**Key differences between the three modes:**

| Mode | Explicit INSERT | NULL INSERT | No value |
|------|----------------|------------|----------|
| `GENERATED ALWAYS` | Error | Error | Auto-generated |
| `GENERATED BY DEFAULT` | Uses your value | Inserts NULL | Auto-generated |
| `GENERATED BY DEFAULT ON NULL` | Uses your value | Auto-generated | Auto-generated |

### 6. SQL Server IDENTITY

```sql
-- SQL Server: IDENTITY column
CREATE TABLE cmic.job_cost_detail (
    entry_id INT IDENTITY(1,1) PRIMARY KEY,
    job_id   VARCHAR(20),
    amount   DECIMAL(15,2)
);

-- Get the last generated value
SELECT SCOPE_IDENTITY();        -- last identity in current scope
SELECT @@IDENTITY;              -- last identity in current session (any scope)
SELECT IDENT_CURRENT('cmic.job_cost_detail');  -- last identity for this table
```

### 7. SQL Server SEQUENCE (2012+)

```sql
-- SQL Server: sequence object (similar to Oracle)
CREATE SEQUENCE cmic.job_cost_seq
    AS INT
    START WITH 1
    INCREMENT BY 1
    CACHE 20;

-- Use in INSERT
INSERT INTO cmic.job_cost_detail (entry_id, job_id, amount)
VALUES (NEXT VALUE FOR cmic.job_cost_seq, 'J-1001', 5000.00);

-- Use as column default
ALTER TABLE cmic.job_cost_detail
    ADD CONSTRAINT df_entry_id DEFAULT NEXT VALUE FOR cmic.job_cost_seq FOR entry_id;
```

## Code Examples

### Creating and Using an Oracle Sequence

```sql
-- Oracle: create sequence for cost entry IDs
CREATE SEQUENCE cmic.cost_entry_seq
    START WITH   100000
    INCREMENT BY 1
    CACHE        100
    NOCYCLE;

-- Insert with sequence
INSERT INTO cmic.job_cost_detail (entry_id, job_id, cost_code, amount, entry_date)
VALUES (cmic.cost_entry_seq.NEXTVAL, 'J-1001', '03-100', 15000.00, SYSDATE);

-- Get current value (only after NEXTVAL in same session)
SELECT cmic.cost_entry_seq.CURRVAL FROM DUAL;

-- Check sequence metadata
SELECT sequence_name, min_value, max_value, increment_by, cache_size, last_number
FROM   all_sequences
WHERE  sequence_owner = 'CMIC';
```

### Resetting a Sequence

```sql
-- Oracle: there is no ALTER SEQUENCE RESTART in pre-18c
-- Workaround: drop and recreate, or use INCREMENT BY trick

-- Step 1: find current value
SELECT cmic.cost_entry_seq.NEXTVAL FROM DUAL;  -- say it returns 50000

-- Step 2: set increment to go backwards
ALTER SEQUENCE cmic.cost_entry_seq INCREMENT BY -49999;

-- Step 3: call NEXTVAL to get back to 1
SELECT cmic.cost_entry_seq.NEXTVAL FROM DUAL;  -- returns 1

-- Step 4: reset increment
ALTER SEQUENCE cmic.cost_entry_seq INCREMENT BY 1;

-- Oracle 18c+: much simpler
ALTER SEQUENCE cmic.cost_entry_seq RESTART START WITH 1;
```

### Oracle Sequence in a Trigger (Pre-12c Pattern)

```sql
-- Oracle: before-insert trigger to auto-populate ID (pre-12c pattern)
CREATE OR REPLACE TRIGGER cmic.trg_cost_detail_bi
    BEFORE INSERT ON cmic.job_cost_detail
    FOR EACH ROW
    WHEN (NEW.entry_id IS NULL)
BEGIN
    :NEW.entry_id := cmic.cost_entry_seq.NEXTVAL;
END;
/
```

### Identity Column — GENERATED ALWAYS

```sql
-- Oracle 12c+: identity column
CREATE TABLE cmic.safety_incidents (
    incident_id NUMBER GENERATED ALWAYS AS IDENTITY,
    job_id      VARCHAR2(20) NOT NULL,
    incident_date DATE NOT NULL,
    description VARCHAR2(4000),
    severity    NUMBER(1)
);

-- This works:
INSERT INTO cmic.safety_incidents (job_id, incident_date, description, severity)
VALUES ('J-1001', SYSDATE, 'Ladder fall near scaffolding', 3);

-- This FAILS with ORA-32795:
INSERT INTO cmic.safety_incidents (incident_id, job_id, incident_date, description, severity)
VALUES (999, 'J-1001', SYSDATE, 'Manual ID attempt', 3);
```

### Migrating a Sequence to SQL Server IDENTITY

```sql
-- Step 1: Find the current max value in Oracle
SELECT MAX(entry_id) FROM cmic.job_cost_detail;
-- Returns: 587432

-- Step 2: Create SQL Server table with IDENTITY starting above the max
CREATE TABLE cmic.job_cost_detail (
    entry_id INT IDENTITY(587433, 1) PRIMARY KEY,
    job_id   VARCHAR(20),
    cost_code VARCHAR(20),
    amount   DECIMAL(15,2),
    entry_date DATETIME2(0)
);

-- Step 3: Load historical data with IDENTITY_INSERT ON
SET IDENTITY_INSERT cmic.job_cost_detail ON;

INSERT INTO cmic.job_cost_detail (entry_id, job_id, cost_code, amount, entry_date)
SELECT entry_id, job_id, cost_code, amount, entry_date
FROM   oracle_staging.job_cost_detail;

SET IDENTITY_INSERT cmic.job_cost_detail OFF;

-- Step 4: Reseed if needed
DBMS CHECKIDENT('cmic.job_cost_detail', RESEED, 587432);
-- Next insert gets 587433
```

### Migrating a Sequence to SQL Server SEQUENCE

```sql
-- Alternative: use SQL Server SEQUENCE (closer to Oracle behavior)
-- Step 1: find current value in Oracle
SELECT cmic.cost_entry_seq.NEXTVAL FROM DUAL;  -- 587433

-- Step 2: create matching SQL Server sequence
CREATE SEQUENCE cmic.cost_entry_seq
    AS INT
    START WITH 587434  -- one above Oracle's current
    INCREMENT BY 1
    CACHE 100;

-- Step 3: use in INSERT
INSERT INTO cmic.job_cost_detail (entry_id, job_id, cost_code, amount)
VALUES (NEXT VALUE FOR cmic.cost_entry_seq, 'J-1001', '03-100', 5000.00);
```

### Bulk Insert with Sequence

```sql
-- Oracle: using sequence in a bulk INSERT ... SELECT
INSERT INTO cmic.job_cost_archive (archive_id, entry_id, job_id, amount)
SELECT cmic.archive_seq.NEXTVAL, entry_id, job_id, amount
FROM   cmic.job_cost_detail
WHERE  entry_date < ADD_MONTHS(SYSDATE, -24);
```

## Common Patterns

### Pattern 1 — Sequence + Trigger (Legacy Oracle)

Pre-12c Oracle universally used a before-insert trigger that called `sequence.NEXTVAL` to
populate the primary key. Every CMiC table likely has this pattern. When migrating, replace
with `IDENTITY` in SQL Server.

### Pattern 2 — Parent-Child with CURRVAL

Use `NEXTVAL` for the parent row, then `CURRVAL` for child rows in the same session:

```sql
INSERT INTO cmic.change_orders (co_id, job_id, amount)
VALUES (cmic.co_seq.NEXTVAL, 'J-1001', 50000);

INSERT INTO cmic.co_line_items (line_id, co_id, description, amount)
VALUES (cmic.co_line_seq.NEXTVAL, cmic.co_seq.CURRVAL, 'Extra excavation', 50000);
```

In SQL Server, use `SCOPE_IDENTITY()` or `OUTPUT` clause instead.

### Pattern 3 — Gapless Sequences (Rare)

Some regulatory or accounting requirements demand gapless numbering (invoice numbers,
check numbers). Oracle sequences **cannot** guarantee gapless IDs because gaps occur on
rollback and cache loss. For gapless sequences, use a serialized counter table with
`SELECT FOR UPDATE` — but this creates a bottleneck.

### Pattern 4 — Multi-Table ID Generation

A single Oracle sequence can serve multiple tables. This is useful when you need globally
unique IDs across tables. SQL Server SEQUENCE objects work the same way.

## Gotchas and Pitfalls

1. **Gaps are guaranteed.** Oracle sequences gap on rollback, instance restart, and RAC
   cache allocation. Do not use sequence values as a proxy for "number of records."

2. **CURRVAL requires prior NEXTVAL.** Calling `seq.CURRVAL` without a prior `NEXTVAL` in
   the same session raises `ORA-08002: sequence not yet defined in this session`.

3. **NOCACHE is slow.** `NOCACHE` writes to the data dictionary on every NEXTVAL call.
   Under high concurrency, this becomes a massive bottleneck. Always use CACHE unless you
   have a specific reason not to.

4. **IDENTITY_INSERT and bulk loads.** When migrating data to a SQL Server IDENTITY column,
   you must enable `SET IDENTITY_INSERT ON` for that table. Only one table per session can
   have it enabled at a time.

5. **GENERATED ALWAYS prevents manual ID.** If the Oracle table uses `GENERATED ALWAYS AS
   IDENTITY`, you cannot insert explicit ID values. You must use `GENERATED BY DEFAULT` if
   you need to override the identity during data loads.

6. **Sequence value overflow.** If a sequence reaches MAXVALUE with NOCYCLE, the next
   NEXTVAL raises an error. Monitor sequence headroom during migration, especially for
   sequences with small MAXVALUE.

7. **DBCC CHECKIDENT quirks.** SQL Server's `DBCC CHECKIDENT` reseeds identity, but the
   behavior differs based on whether the table is empty or populated. Test reseed behavior
   before relying on it.

8. **ORDER option in RAC.** In Oracle RAC, `ORDER` forces sequences to be globally ordered
   across instances, which reduces concurrency. Only use `ORDER` when strict ordering is
   required (rare in data engineering).

9. **Negative increment sequences.** Oracle allows `INCREMENT BY -1` for descending
   sequences. SQL Server SEQUENCE also supports this, but IDENTITY does not support
   negative increments.

## Performance Considerations

- **CACHE size tuning**: Higher cache = fewer data dictionary writes = better performance
  under concurrency. For high-volume tables (cost entries, timecards), use `CACHE 1000` or
  higher. The trade-off is larger gaps on restart.

- **Sequence contention in RAC**: In Oracle RAC, each instance gets its own cache range.
  Instance 1 might generate 1–1000, instance 2 gets 1001–2000. Values are not in global
  order. Use `ORDER` only if ordering matters (adds contention).

- **IDENTITY vs SEQUENCE performance in SQL Server**: IDENTITY is slightly faster than
  SEQUENCE for single-row inserts because it avoids the separate object lookup. For bulk
  inserts, the difference is negligible.

- **Trigger overhead**: The pre-12c trigger-based pattern adds overhead to every INSERT.
  Oracle 12c identity columns eliminate this overhead. When migrating to SQL Server
  IDENTITY, you get the same benefit.

- **Bulk insert with sequences**: `INSERT ... SELECT` with `seq.NEXTVAL` in Oracle
  performs well because the sequence is called once per row in a set operation. In SQL
  Server, `IDENTITY` handles this automatically.

- **Sequence pre-allocation for ETL**: If you know you are about to insert 1 million rows,
  you can pre-allocate by calling NEXTVAL in a loop or adjusting the sequence with
  `ALTER SEQUENCE INCREMENT BY 1000000; SELECT seq.NEXTVAL FROM DUAL; ALTER SEQUENCE
  INCREMENT BY 1;`. This avoids mid-load cache refills.

## BNBuilders Context

CMiC ERP at BNBuilders uses Oracle sequences extensively:

- **Job cost entry IDs**: Every cost transaction in CMiC gets its ID from a sequence.
  During migration, capture the current sequence value and start the SQL Server IDENTITY
  above it to prevent ID collisions.

- **Change order numbering**: Change orders have sequential IDs generated by sequences.
  Project managers rely on these numbers for communication. Ensure continuity during
  migration.

- **Equipment log IDs**: Equipment checkout/return records use sequences. High-volume
  during construction season — optimize cache size.

- **Timecard entries**: Field workers submit timecards daily. Each entry gets a
  sequence-generated ID. Peak volumes during summer months can be thousands per day per
  project. Use `CACHE 500` or higher on the Oracle side.

- **Migration strategy**: For each CMiC table with a sequence:
  1. Query `MAX(id)` from the Oracle table
  2. Create SQL Server table with `IDENTITY(max_id + 1, 1)`
  3. Bulk load historical data with `IDENTITY_INSERT ON`
  4. Verify the identity seed with `DBCC CHECKIDENT`
  5. Test inserts to confirm new IDs continue the sequence

- **Dual-write period**: During cutover, both Oracle and SQL Server may be receiving
  inserts. Coordinate sequence ranges to prevent collisions. For example, Oracle uses
  odd numbers, SQL Server uses even numbers, then consolidate after cutover.

## Interview / Senior Dev Questions

1. **Q: Why do Oracle sequences produce gaps, and why is this acceptable?**
   A: Gaps occur from rollbacks (sequence does not roll back with the transaction), cache
   loss on instance restart, and RAC cache allocation. This is acceptable because
   sequences are designed for uniqueness, not contiguity. Gapless numbering requires a
   different mechanism (serialized counter table) that sacrifices concurrency.

2. **Q: You are migrating a CMiC table with 5 million rows and a sequence-generated PK.
   How do you handle the ID column in SQL Server?**
   A: Find the MAX(id) in Oracle. Create the SQL Server table with
   `IDENTITY(max_id + 1, 1)`. Enable `IDENTITY_INSERT ON`, bulk load the data preserving
   original IDs, then disable `IDENTITY_INSERT`. Verify with `DBCC CHECKIDENT`. New
   inserts in SQL Server will continue where Oracle left off.

3. **Q: What is the difference between GENERATED ALWAYS and GENERATED BY DEFAULT in Oracle
   12c identity columns?**
   A: `GENERATED ALWAYS` never allows manual ID insertion — Oracle always generates the
   value. `GENERATED BY DEFAULT` generates a value only when the user does not supply one.
   `GENERATED BY DEFAULT ON NULL` generates a value when NULL is explicitly passed. For
   data migration, `BY DEFAULT` is more flexible because you can load historical IDs.

4. **Q: When would you choose SQL Server SEQUENCE over IDENTITY for a migration?**
   A: When you need the ID before the INSERT (parent-child pattern with CURRVAL), when
   multiple tables share one sequence, when you need to control caching, or when the
   Oracle code heavily depends on NEXTVAL/CURRVAL semantics and you want a minimal-change
   migration.

## Quiz

### Question 1
What happens if you call `sequence.CURRVAL` in Oracle without first calling
`sequence.NEXTVAL` in the same session?

<details>
<summary>Show Answer</summary>

You get `ORA-08002: sequence <name> is not yet defined in this session`. CURRVAL only
works after NEXTVAL has been called at least once in the current session. It returns the
last value generated by NEXTVAL in that session — it does not show values generated by
other sessions.
</details>

### Question 2
An Oracle sequence is created with `CACHE 100`. After 37 inserts, the database instance
crashes and restarts. What value will the next `NEXTVAL` call return?

<details>
<summary>Show Answer</summary>

It will return 101 (assuming it started at 1). When the instance started, it cached values
1–100 in memory. After the crash, those cached values (38–100) are lost. On restart, the
sequence allocates the next cache range (101–200), so the next NEXTVAL returns 101. Values
38–100 are permanently skipped — this is one source of gaps in sequences.
</details>

### Question 3
You need to migrate an Oracle table to SQL Server and preserve the original IDs. The table
uses `IDENTITY` in SQL Server. What steps are required?

<details>
<summary>Show Answer</summary>

1. Enable explicit ID insertion: `SET IDENTITY_INSERT schema.table ON;`
2. Include the identity column in your INSERT column list and supply the original values
3. After the load, disable it: `SET IDENTITY_INSERT schema.table OFF;`
4. Reseed if needed: `DBCC CHECKIDENT('schema.table', RESEED, max_id);`
5. Verify the next auto-generated ID is correct by doing a test insert

Note: only one table per session can have IDENTITY_INSERT enabled at a time.
</details>

### Question 4
What is the difference between `SCOPE_IDENTITY()`, `@@IDENTITY`, and
`IDENT_CURRENT('table')` in SQL Server?

<details>
<summary>Show Answer</summary>

- `SCOPE_IDENTITY()` — Returns the last identity value generated in the current scope
  (current stored procedure, trigger, or batch). This is the safest option.
- `@@IDENTITY` — Returns the last identity value generated in the current session, across
  any scope. If a trigger inserts into another table with an identity column, `@@IDENTITY`
  returns that trigger's identity, not yours.
- `IDENT_CURRENT('table')` — Returns the last identity value generated for a specific
  table, regardless of session or scope. Can return values from other users' sessions.

Use `SCOPE_IDENTITY()` for most migration and application code.
</details>
